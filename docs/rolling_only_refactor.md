# Rolling-only refactor

Базовый коммит, под который подготовлен пакет: `f7bfe1b18b9fd264a1b064e19defcfcca851e165`.

## Что изменено

Робот теперь имеет один signal-контур:

`complete BID/ASK bar -> rolling window -> historical candidates -> Pearson -> ranking -> weighted potential -> MARKET signal`.

Удалены сервис подготовки признаков, режим окна SLOT, рыночные режимы, SMA/MA-зоны, уровни вокруг цены, limit-policy сигнала и механизм повторного удержания убыточной позиции.

Оставлены независимые защитные механизмы: обычные TP/SL, контроль зависших защитных ордеров, аварийное MARKET-закрытие при потере защиты, дневной take-profit, дневной flat фьючерсов, rollover-защита, синхронизация позиций, IB clock guard и reconciliation исполнения.

При старте execution-сервис отдельно проверяет брокера на старые `_EXT_TP`/`_EXT_SL` ордера удалённого механизма. Он подтверждённо отменяет их и повторно перечитывает open orders. Если состояние нельзя доказать, execution не запускается — невидимый вспомогательный ордер опаснее отказа старта.

## Почему `mid_close` не добавлен в price DB

`mid_close` однозначно вычисляется как округлённое среднее `bid_close` и `ask_close`. Физический столбец создал бы второй источник истины и потребовал бы синхронно менять history writer, realtime writer, backfill и миграции существующих файлов.

Новый слой `core/price_source.py` строит SQL-выражение `mid_close` при чтении. Signal-контур получает прежнее значение без дублирования данных и без изменения канонической схемы price DB. Candidate search ограничивает основную таблицу сырым диапазоном `bar_time_ts`, поэтому SQLite использует индекс первичного ключа, а не сканирует многолетнюю базу целиком.

## Миграции SQLite

При первом запуске signal-сервис автоматически сужает таблицу `signal_events` до актуального rolling-контракта. Идентификаторы сигналов и прежняя верхняя граница AUTOINCREMENT сохраняются, поэтому удалённый legacy `signal_id` никогда не будет повторно выдан новому сигналу. Из старой таблицы переносятся только разрешённые rolling-записи с выключенным фильтром режима.

При первом обращении к `trade.sqlite3` таблица `trade_intents` автоматически перестраивается без полей старых рыночных режимов и MA-зон. `trade_intent_id`, верхняя граница AUTOINCREMENT и вся операционная история сохраняются. Поля исполнения и защитных ордеров остаются; новые сигнальные intents создаются только как MARKET.

## Безопасное применение

1. Остановить все сервисы робота.
2. Самый безопасный вариант — применять обновление в состоянии FLAT. Первый execution-start дополнительно проверит и отменит старые вспомогательные exit-ордера; при неоднозначном broker state сервис аварийно остановится.
3. Сделать резервные копии `data/state.sqlite3`, `data/trade.sqlite3` и рабочих price DB.
4. Переключиться на ветку рефакторинга и установить актуальные зависимости.
5. Проверить изменения:

   ```powershell
   git diff --check
   python scripts\verify_rolling_refactor.py .
   python -m unittest tester.rolling_signal_smoke_tester tester.rolling_price_pipeline_tester -v
   ```

6. Просмотреть diff перед merge.
7. Первый запуск выполнить на paper account и проверить signal/trader/execution logs.

Новый порядок managed-сервисов:

`market_data -> position_sync -> execution -> signal -> trader`.

Старые отдельные файлы с подготовленными признаками пакет не удаляет из `data/`: это runtime-данные, а не исходный код. После резервной копии их можно архивировать вручную, когда новый контур подтверждён на paper account.
