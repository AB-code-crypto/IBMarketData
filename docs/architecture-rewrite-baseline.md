# Точка начала архитектурного переписывания

**Статус:** зафиксировано  
**Дата:** 2026-07-22  
**Исходный commit:** `5073dbea3f96740b4390e113d06578a514918f00`

**Рабочая ветка:** `agent/architecture-rewrite`

## Назначение

Этот commit отмечает начало полного архитектурного переписывания IBMarketData.

Текущая реализация остаётся исходным материалом и спецификацией фактического поведения: торговой логики, интеграции с Interactive Brokers, хранения данных, восстановления состояния, контроля риска, исполнения ордеров, мониторинга и эксплуатации.

Цель следующего этапа — не косметический рефакторинг существующей структуры, а проектирование и последовательная реализация новой архитектуры с явными границами ответственности, владельцами данных, контрактами между компонентами и переносимыми IB-модулями.

## Что изменяет этот commit

Только документацию и границу истории.

Этот commit намеренно не меняет:

- торговую логику;
- runtime-код;
- схему баз данных;
- конфигурацию;
- deployment;
- поведение сервисов;
- подключение к Interactive Brokers.

## Правила переписывания

1. Все архитектурные изменения выполняются в `agent/architecture-rewrite` или в дочерних ветках от неё.
2. Ветка `baseline/pre-architecture-rewrite-5073dbe` не изменяется, не удаляется и не переписывается force-push.
3. Существующая реализация не удаляется до достижения проверяемого функционального паритета новой системы.
4. Текущее поведение сначала инвентаризируется и фиксируется тестами; только после этого проектируется замена.
5. Каждый этап должен иметь явные входы, выходы, владельца данных, критерии готовности и способ отката.
6. Новая система не переводится в production до завершения архитектурной проверки, автоматических тестов, paper-trading проверки и отдельного плана миграции.
7. Совпадение имён модулей или таблиц со старой системой не считается обязательным. Обязательны сохранение нужного бизнес-поведения и устранение ошибочных связей ответственности.

## План верхнего уровня

1. Инвентаризация текущего проекта и фактического поведения.
2. Карта компонентов, данных, зависимостей и пересекающихся зон ответственности.
3. Формализация неизменяемых бизнес-требований и торговых инвариантов.
4. Проектирование целевой архитектуры и контрактов между компонентами.
5. Последовательная реализация новой системы вертикальными срезами.
6. Сравнение старой и новой реализации на тестах и записанных сценариях.
7. Paper trading, миграция данных, план переключения и план возврата.
8. Удаление старой реализации только после подтверждённого перехода.

## Точка возврата

Безопасная исходная версия доступна двумя способами:

```bash
git switch baseline/pre-architecture-rewrite-5073dbe
```

или непосредственно по commit:

```bash
git switch --detach 5073dbea3f96740b4390e113d06578a514918f00
```

Для локального принудительного возврата ветки к исходной точке, только после сохранения нужных изменений:

```bash
git reset --hard 5073dbea3f96740b4390e113d06578a514918f00
```

`reset --hard` уничтожает незакоммиченные локальные изменения и не должен использоваться как обычный способ переключения между версиями.

## Критерий успеха переписывания

Переписывание считается успешным не тогда, когда новый код выглядит чище, а когда одновременно выполнены условия:

- зоны ответственности не пересекаются без явного контракта;
- каждый изменяемый набор данных имеет одного владельца записи;
- IB-интеграционные модули отделены от логики конкретного робота там, где это оправдано;
- торговые и риск-инварианты формализованы и покрыты тестами;
- восстановление после рестарта и сбоев проверено;
- старая и новая системы дают ожидаемые результаты на контрольных сценариях;
- существует проверенный план переключения и возврата.
